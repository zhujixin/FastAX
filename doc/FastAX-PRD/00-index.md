## 3. 产品概述

FastAX 是一个 **Token 代理与交易平台**，作为连接终端用户与 Token 源头（官方平台、API 供应商、第三方 Token 厂家）的中间枢纽，提供统一、安全、高效、合规的 Token 中转、交易与增值服务。平台不仅自营 Token 代理，还开放第三方厂家入驻，打造 Token 交易市场。

### 2.2 产品背景

随着数字经济与区块链、AI 大模型技术的深度融合，Token 作为数字资产与算力资源的核心载体，已覆盖供应链金融、版权管理、AI 模型调用、碳交易等多个领域。当前用户在使用 Token 过程中面临以下痛点：

| 痛点 | 描述 |
|------|------|
| 海外调用门槛高 | 海外 Token 调用需外币支付，网络不稳定，延迟高 |
| **国内模型海外访问难** | **海外用户访问国内大模型（DeepSeek、通义千问等）缺乏优化链路，延迟高、文档不全** |
| 多平台接口不统一 | 各家 Token 供应商接口各异，集成繁琐 |
| 账号风控风险 | 个人账号易被风控，影响业务连续性 |
| 安全隐患 | Token 存储与传输存在泄露风险 |
| 合规管控难 | 多国政策差异，缺乏统一的合规兜底 |
| **供应商销售渠道单一** | **中小 Token 厂家缺乏面向开发者群体的直接销售渠道和支付能力** |

### 2.3 产品范围

**包含范围**：

- Token 多源头对接与渠道管理（含国内大模型供应商）
- **第三方 Token 厂家入驻与自主销售（店铺管理、商品上架、自主定价、结算提现）**
- Token 代理中转与调用（支持流式输出，含海外→国内专线优化）
- Token 商城与交易系统
- 用户与账号管理（含企业子账号，支持邮箱注册）
- 风控引擎（异常检测、防刷、反洗钱）
- 统计分析、运营工具与通知系统
- 管理后台（用户、Token、交易、风控、系统配置）
- **多语言国际化（i18n）**：前端界面、通知、文档的多语言支持（详见第 6.9 节）

**不包含范围**：

- 不直接发行或铸造 Token（仅作为代理中转层）
- 不提供区块链底层节点服务
- 不涉及加密货币交易所功能（仅 Token 代理买卖）

### 2.4 核心价值主张

| 价值 | 说明 |
|------|------|
| **统一接入** | 一套接口对接全球主流 Token 供应商及第三方厂家，兼容 OpenAI 协议 |
| **开放市场** | 第三方 Token 厂家可自主入驻、上架商品、灵活定价，平台提供交易撮合与结算 |
| **双向稳定** | 海外→国内 / 国内→海外双轨专线 + 智能路由，目标可用率 ≥ 99.5% |
| **安全合规** | 加密存储、AI 风控、反洗钱检测、合规审计日志 |
| **透明可控** | 价格透明、用量统计、消费记录、账单导出 |

## 4. 产品目标与成功指标

### 3.1 产品目标

1. **MVP 目标（上线 3 个月内）**：支撑 1000+ 注册用户，日活 200+，日均 Token 调用量 10 万次
2. **增长目标（上线 6-12 个月）**：注册用户 1 万+，月交易额 100 万+
3. **质量目标**：系统可用性 ≥ 99.5%，核心接口延迟 ≤ 500ms，Token 中转成功率 ≥ 99%
4. **多语言目标**：MVP 支持中英双语，6 个月内扩展至 4 种语言，12 个月内扩展至 6+ 语言（详见第 14 章）
5. **海外用户目标（上线 6 个月内）**：海外用户可通过 FastAX 稳定调用国内主流大模型（DeepSeek、通义千问、智谱 GLM 等），延迟 ≤ 1500ms

### 3.2 关键成功指标 (KPI)

| 指标 | 目标值 |
|------|--------|
| 系统可用率 | ≥ 99.5% |
| Token 中转延迟（P95） | ≤ 1000ms |
| 流式输出首包延迟 | ≤ 500ms |
| 注册转化率 | ≥ 40% |
| 用户月留存率 | ≥ 60% |
| Token 调用成功率 | ≥ 99% |
| 交易成功率 | ≥ 99.9% |
| 安全事件响应时间 | ≤ 15 分钟 |
| 风控误杀率 | ≤ 1% |
| 合规审计日志覆盖率 | 100% |
| **核心页面翻译覆盖率** | **≥ 95%（MVP）** |
| **语言切换响应时间** | **≤ 200ms** |
| **翻译文件对首屏性能影响** | **≤ 50ms** |
| **海外用户注册转化率提升** | **≥ 30%（相比无多语言版本）** |
| **多协议兼容度** | **Anthropic/Gemini/OpenAI 原生协议 100% 覆盖（P0 阶段）** |
| **安全护栏检测率** | **PII 检测率 ≥ 99%，Prompt 注入检测率 ≥ 95%，误报率 ≤ 2%** |

## 5. 用户角色与画像

### 4.1 角色矩阵

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| **普通用户** | 个人开发者、小型团队，直接购买和使用 Token | 便捷购买、稳定调用、价格透明 |
| **企业用户** | 中型以上企业，需管理多个子账号和 Token 配额 | 子账号管理、额度分配、账单对账 |
| **代理/分销用户** | 以分销模式推广平台 Token 的合作伙伴 | 佣金体系、推广链接、客户管理 |
| **海外用户** | 非中文使用者，通过海外渠道访问平台，使用国内大模型（DeepSeek、通义千问等） | 英文界面、Stripe 支付、国内模型调用、多语言文档 |
| **Token 厂家（供应商）** | 第三方 Token 生产方或转售方，在平台注册开店自主销售 | 店铺管理、商品上架、自主定价、销售统计、结算提现 |
| **平台管理员** | 运营/运维团队，负责平台日常管理 | 用户审核、Token 管控、供应商审核、交易审核、风控、系统配置 |
| **超级管理员** | 系统所有者，拥有全部权限 | 系统参数配置、管理员账号管理、审计日志 |

### 4.2 用户画像

#### 画像 1：个人开发者 — 小张

- **背景**：25 岁，全栈开发者，独立做 AI 应用开发
- **痛点**：海外 API 调用不稳定，个人账号经常被风控，每个月要管理多个平台的 API Key
- **期望**：一个账号搞定所有模型调用，延迟低、价格透明，不用操心账号被封
- **使用场景**：通过 FastAX 中转调用 OpenAI、Claude 的 API，在个人项目中使用

#### 画像 2：企业技术负责人 — 李经理

- **背景**：38 岁，某 AI 创业公司技术总监，团队 20 人
- **痛点**：团队每个人各自管理 API Key，Token 消耗无法统一管控，月度对账困难
- **期望**：统一的企业账号，子账号额度分配，详细的用量报表和账单
- **使用场景**：公司统一在 FastAX 采购 Token，各团队使用子账号调用，月度统一对账

#### 画像 3：海外开发者 — Tom

- **背景**：美国全栈开发者，习惯使用英文界面和技术文档，关注 DeepSeek、通义千问等国内高性价比模型
- **痛点**：中文平台无法使用，缺少英文界面和支付方式；直接访问国内大模型延迟高、文档不全
- **期望**：全英文界面，Stripe 美元支付，低延迟访问国内大模型，英文技术文档
- **使用场景**：通过 FastAX 中转调用 DeepSeek、Qwen 等国内模型 API，在个人和商业项目中使用

#### 画像 4：Token 厂家 — 陈总

- **背景**：42 岁，某中小 AI 模型公司创始人，团队自研了垂直领域大模型
- **痛点**：自有模型缺乏面向海外开发者的销售渠道；对接支付、部署、客服成本高
- **期望**：快速入驻平台，上架模型并自主定价，专注模型优化，销售和交付交给平台
- **使用场景**：在 FastAX 注册供应商账号，上传模型 API 信息，设置价格和配额，通过平台获取全球客户

#### 画像 5：平台运营 — 王运营

- **背景**：30 岁，互联网平台运营，负责 FastAX 日常运营
- **痛点**：需要及时发现异常交易和恶意用户，同时要保证正常用户体验
- **期望**：强大的风控后台、数据看板、运营工具
- **使用场景**：日常监控平台数据，处理用户审核和投诉，配置运营活动

```

| 阶段 | 用户故事 | 优先级 |
|------|----------|--------|
| **注册登录** | 作为用户，我希望通过手机号/邮箱快速注册，以便开始使用平台 | P0 |
| | 作为用户，我希望能通过微信/Google 等第三方账号登录，方便快捷 | P1 |
| **浏览商城** | 作为用户，我希望查看所有可用的 Token 类型和价格，以便做出购买决策 | P0 |
| | 作为用户，我希望能按模型/供应商筛选 Token，快速找到需要的类型 | P1 |
| **下单购买** | 作为用户，我希望能选择 Token 数量和支付方式，完成购买 | P0 |
| **支付** | 作为用户，我希望能通过微信/支付宝/银行卡完成支付，支付后 Token 实时到账 | P0 |
| **Token 调用** | 作为开发者，我希望能通过标准 OpenAI 兼容接口调用 Token，无需修改代码 | P0 |
| | 作为用户，我希望调用延迟低、稳定性高，不影响业务 | P0 |
| **用量查询** | 作为用户，我希望查看我的 Token 使用记录和消费明细，以便管理预算 | P1 |
| | 作为用户，我希望导出账单，用于财务报销 | P2 |
| **多语言切换** | 作为海外用户，我希望页面内容以我的语言展示，以便顺畅使用平台 | P0 |
| | 作为开发者，我希望 API 错误消息使用我的语言，便于调试 | P1 |
| **国内模型调用** | 作为海外开发者，我希望能通过统一接口调用 DeepSeek、Qwen 等国内大模型，无需单独注册 | P0 |
| | 作为海外开发者，我希望访问国内模型的延迟在可接受范围内（≤ 1500ms） | P0 |
| | 作为海外用户，我希望通过 Stripe 等国际支付方式购买 Token | P0 |
| **供应商入驻** | 作为 Token 厂家，我希望在平台注册并自助上架 Token 商品，快速面向全球销售 | P0 |
| | 作为供应商，我希望自主设置商品价格和折扣策略 | P0 |

```

风险监控 → 系统设置
