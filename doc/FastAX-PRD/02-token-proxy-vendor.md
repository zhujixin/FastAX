#### 6.2.7 转发逻辑设计（ROUTE）

综合参考 [CC Switch](https://www.ccswitch.co/) 和 [one-api](https://github.com/songquanpeng/one-api) 的设计理念，FastAX 转发层需实现智能路由、故障转移、负载均衡等核心能力。

**路由策略**：采用 one-api 验证的**优先级分组 + 同优先级权重随机**方案，放弃实时加权评分公式。
渠道选择流程：`group+model 查 Ability 索引表(或内存缓存) → 按 priority 分组 → 最高优先级组内 rand 随机 → 健康检查过滤`。

**熔断策略**：采用 one-api 轻量级自动禁用方案（不依赖 gobreaker 状态机），对 5xx/超时错误自动禁用渠道，通过健康检测恢复。排除 400/401/403/429 等非渠道故障。

**重试策略**：参考 one-api `Relay()` 函数实现，失败时自动跨渠道重试（跳过已失败的渠道），仅对 429/5xx/超时重试。

| 功能 | 需求描述 | 优先级 | 备注 |
|------|----------|--------|------|
| ROUTE-01 | **渠道选择算法（Ability 索引）**：基于 group+model 查询 Ability 索引表或内存缓存获取匹配渠道列表，按优先级分组，同优先级组内按权重随机选择，再经健康检查过滤后返回最优渠道 | P0 | 参考 one-api CacheGetRandomSatisfiedChannel，O(1) 查询 |
| ROUTE-02 | **加权随机选择**：在最高优先级组内按权重比例随机选择，权重高的渠道被选中概率更大 | P0 | 替代加权轮询，更适合短连接 API 调用 |
| ROUTE-03 | **自动故障转移（Failover）** + **重试**：主供应商失败时自动切换渠道重试（跳过刚失败的渠道），可配置重试次数（默认 3 次） | P0 | 参考 one-api controller/relay.go 重试循环 |
| ROUTE-04 | **轻量熔断**：对 5xx/超时错误自动禁用渠道（DB 更新 status=3），排除 400/401/403/429；自动禁用后通过健康检测恢复 | P0 | 参考 one-api monitor.ShouldDisableChannel，不用 gobreaker |

| ROUTE-06 | **延迟敏感路由**：根据用户地域自动选择最低延迟的节点/供应商 | P0 | 海外用户→海外节点→国内供应商最优路径 |
| ROUTE-07 | **成本优化路由**：支持"优先低价"模式，在满足延迟要求的前提下自动选择成本最低的供应商 | P1 | — |
| ROUTE-08 | **时段性调度**：支持按时间段配置不同路由策略（如非工作时段切至低成本渠道） | P1 | — |
| ROUTE-09 | **请求重写与透明转发**：转发请求时自动改写端点路径、Header，保持与上游 API 的兼容 | P0 | 参考 one-api SetupContextForSelectedChannel |
| ROUTE-10 | **模型自动发现**：通过 `/v1/models` 自动获取供应商可用模型列表，按模型分组展示 | P1 | 供应商新增模型时无需手动配置 |

| ROUTE-12 | **流式请求故障转移**：流式输出（SSE）过程中上游断开时，在当前请求内重试或切换备用供应商 | P0 | 关键！流式场景下的高可用保障 |
